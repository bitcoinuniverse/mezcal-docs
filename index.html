Runes-inspired assets that place a JSON Mezcalstone directly after OP_RETURN. No binary Runestone decoder, no etching commitment, and strict deterministic validation.

The difference

JSON first.
Bitcoin native.

Mezcal keeps the UTXO allocation model familiar from Runes, then replaces compact Runestone decoding with a strict JSON message. That makes the protocol easier to inspect, but not looser to validate.

01 / CARRIER

Directly after OP_RETURN

The official README specifies an OP_RETURN followed by a UTF-8 encoded Mezcalstone string. There is no OP_13 marker.

02 / STATE

UTXO allocations

Etchings, mints, edicts, and a pointer determine the asset amounts allocated to the transaction outputs.

03 / VALIDATOR

Strict schema, cenotaph on failure

The reference indexer parses JSON and validates it against a strict schema. Invalid JSON, fields, or edict outputs create a cenotaph.

Lifecycle

From etch to output.

Build the transaction outputs before you derive edict indexes. The visible JSON is only half the protocol, the final transaction shape completes the allocation.

01

Etch

Choose one unique lowercase name, symbol, divisibility, premine, and optional mint terms.

02

Mint

Reference an existing block:tx asset ID and satisfy caps, windows, and price terms.

03

Allocate

Use edicts to name output indexes. Use a pointer for remaining, unallocated amount.

04

Verify

Decode the finalized OP_RETURN payload and compare the indexed per-output result after confirmation.
